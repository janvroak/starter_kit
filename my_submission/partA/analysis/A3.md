## Conceptual Issue: The Metric, Not the Code

The implementation correctly computes **tokens per whitespace word**, but this is not necessarily the most appropriate quantity for multilingual routing or cost estimation.

Whitespace-delimited words are not directly comparable across writing systems. Languages differ in how information is segmented into words, meaning that "one word" does not represent a constant amount of information across languages.

As a result, even a perfectly implemented Tokens per Word metric can introduce systematic bias when comparing languages.

This is not a software bug. The implementation behaves exactly as intended. The conceptual problem is that the chosen denominator (whitespace words) is not invariant across languages, making it a poor choice for routing and cost estimation in multilingual systems.

This hypothesis was tested by comparing alternative normalization strategies (Tokens per Character and Tokens per Byte) in A3.



### Which Single Metric Should Drive Routing?

For routing and cost estimation, the normalization metric should keep approximately the same quantity constant across languages.

Tokens per Word is easy to interpret but depends heavily on language-specific word segmentation rules.

Tokens per Character removes dependence on whitespace but still treats characters from different writing systems equally despite differences in Unicode representation.

Tokens per Byte provides the most language-independent normalization among the evaluated metrics because it measures tokenization relative to the encoded text processed by the tokenizer. Although UTF-8 uses different numbers of bytes for different scripts, the metric avoids assumptions about word boundaries and is therefore more suitable for comparing multilingual inputs.

For these reasons, **Tokens per Byte** is recommended as the primary metric for routing and cost estimation, while Tokens per Word remains useful for human interpretation.

## Discussion

The larger evaluation corpus confirms the trend observed in the sample corpus.

GPT-2 produces substantially more tokens for Indic languages than for English, with Kannada and Tamil showing particularly high fertility. However, this behaviour changes significantly when using XLM-Roberta.

Under XLM-Roberta, the token counts become much more balanced across all four languages. This indicates that the observed differences are largely due to tokenizer design and vocabulary coverage rather than intrinsic properties of the languages themselves.

Therefore, conclusions about language cost should not be made using a tokenizer that is not intended for multilingual text.

## Comparison of Normalization Metrics

Three normalization strategies were evaluated.

### Tokens per Word

Advantages:
- Easy to interpret.
- Commonly used in tokenizer analysis.

Limitations:
- Depends on language-specific word segmentation.
- Languages differ in how words are written and separated.

### Tokens per Character

Advantages:
- Independent of whitespace.
- More stable across writing systems.

Limitations:
- Character counts differ between scripts and do not directly correspond to storage or inference cost.

### Tokens per Byte

Advantages:
- Independent of language-specific word boundaries.
- Closely related to the amount of encoded text processed by the tokenizer.
- Allows fairer comparisons across scripts using different Unicode encodings.

Limitations:
- Less intuitive than words.
- UTF-8 uses different numbers of bytes for different scripts.

---

## Recommendation

For engineering applications such as routing requests to language models, tokenizer behaviour should always be measured using the tokenizer that will actually be deployed.

Among the evaluated metrics, Tokens per Byte provides the most language-independent normalization, while Tokens per Word remains useful for interpretability.

The experiments also demonstrate that tokenizer choice has a much larger effect on fertility than language alone. Consequently, deployment decisions should prioritise selecting an appropriate multilingual tokenizer rather than assuming that all languages incur similar tokenization costs.
